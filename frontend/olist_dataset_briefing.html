What is this dataset?

100KReal Orders
9Linked Tables
3 yrs2016 – 2018

The Brazilian E-Commerce Public Dataset by Olist is a real-world, anonymized transactional dataset covering orders placed across multiple Brazilian marketplaces operated by Olist — the largest department store network in Brazil.

Olist connects small businesses across Brazil to channels through a single contract. After a customer purchases, the seller is notified to fulfill; after delivery (or estimated delivery date), the customer receives a satisfaction survey.

All company and partner references in review texts have been replaced with names of Game of Thrones great houses to preserve anonymity.

Key characteristics

  • Real commercial data — not synthetic, not sampled from a toy source
  • Multi-dimensional: view each order via status, price, payment, freight, location, product, and review
  • Relational: 9 CSV files linked by order and customer keys — natural SQL join practice
  • Geographic: includes a geolocation table mapping Brazilian zip codes to lat/lng
  • Joinable with the Olist Marketing Funnel dataset for end-to-end funnel analysis
  • Multilingual: product categories in Portuguese, with a translation reference table

How to utilize it

Getting started

  • Download all 9 CSVs from kaggle.com/datasets/olistbr/brazilian-ecommerce
  • The orders table is the spine — always load it first, join outward
  • Join order_items → products for product-level analysis
  • Join customers or sellers → geolocation via zip code for geo analysis
  • Join category_name_translation to products for readable English categories

Recommended loading order

1. olist_customers_dataset.csv        -- foundation
2. olist_sellers_dataset.csv
3. olist_products_dataset.csv
4. product_category_name_translation.csv
5. olist_orders_dataset.csv           -- spine
6. olist_order_items_dataset.csv
7. olist_order_payments_dataset.csv
8. olist_order_reviews_dataset.csv
9. olist_geolocation_dataset.csv      -- enrichment

Platforms you can load it into

  • DuckDB — fastest for local CSV analytics, zero setup
  • PostgreSQL / MySQL — standard SQL engines for relational testing
  • Apache Spark / Databricks — for PySpark and Delta Lake exercises
  • BigQuery / Snowflake / Redshift — cloud warehouse testing
  • Pandas / Polars — Python dataframe exercises
LLM / Claude Code tip

Always pass the schema DDL (CREATE TABLE statements with column names and types) as system context when prompting any of the 6 tools. This alone dramatically improves join path inference and query generation quality.

Use order_id as your primary join anchor. Every fact table connects back to it. For Claude Code agents, a one-paragraph schema summary + the DDL is sufficient — no need to paste sample rows.

Tool-by-tool mapping

1
sql-agent
SQL & PySpark Coding Agent
Ideal fit ▼
Why it works
  • 9 normalized tables = rich multi-join SQL exercises across a realistic schema
  • Aggregations: SUM(payment_value), AVG(review_score), COUNT(DISTINCT customer_id)
  • Time-series: order_purchase_timestamp spans 2016–2018 for trend queries
  • Geo-queries: lat/lng from geolocation table enables distance and region filtering
  • Filter/group combinations: by state, product category, payment type, seller, review score
Sample prompts
  • What are the top 5 states by total revenue?
  • Which product categories have the worst average review scores?
  • Show monthly order volume trend for 2017
  • Find sellers with more than 100 orders and less than 3.5 avg review score
  • What percentage of orders were delivered late vs estimated date?
Purely analytical / read-only. No stored procedures, triggers, or live transactional writes to test.
2
meeting-intelligence
Meeting Intelligence Agent
Indirect fit ▼
Why it works (indirectly)
  • order_reviews free-text comments are unstructured, sentiment-laden natural language — good NLP proxy
  • Reviews contain product complaints, delivery feedback, and praise — solid for entity extraction and sentiment testing
  • Portuguese text with English translation table = multilingual NLP capability testing
  • Review reply fields simulate a two-party dialogue (customer + seller response)
How to frame it
  • Treat reviews as "customer feedback call transcripts"
  • Generate synthetic meeting notes derived from order data (e.g., weekly seller performance review)
  • Use seller + review + order data to simulate a sales call debrief
Not actual meeting transcripts. You'll need to reframe or generate synthetic meeting text from the dataset to properly exercise meeting intelligence features.

Data Architecture Advisor
Good fit ▼
Why it works

Sample architecture questions

  • How would you partition the orders table in Spark?
  • Propose a medallion architecture for this dataset
  • What are the SCD type 2 candidates in this schema?
Static dataset — no CDC (change data capture) or streaming scenarios to stress-test real-time architecture advice.

pipeline-designer
Multi-Agent Pipeline Designer
Good fit ▼
Why it works
  • Natural ETL pipeline: raw CSVs → cleaned → joined → aggregated → dashboard-ready
  • Multiple transformation stages: deduplication of geolocation, currency normalization, category translation
  • Tests multi-step orchestration: ingest → validate → transform → load → serve
  • Data quality checks: null delivery dates, duplicate zip codes, missing review scores
Sample pipeline prompts
  • Design a pipeline to ingest all 9 CSVs, join them, and produce a daily sales summary
  • Add data quality checks for null order_delivered_customer_date values
  • Build a pipeline that flags orders with review_score below 2 for a support team
  • How would you handle late-arriving payment records in this schema?
All batch data — no real-time or event-driven records. Won't stress-test streaming pipeline design well without simulating micro-batches.

databricks-agent
Databricks Intelligence Agent
Strong fit ▼
Why it works
  • 100K rows = fast iteration in notebooks, but schema complexity simulates production workloads
  • PySpark exercises: joins, window functions, UDFs for geo-distance calculation
  • Delta Lake medallion: bronze (raw CSVs) → silver (cleaned joins) → gold (aggregated KPIs)
  • MLflow angle: train a delivery delay predictor or review score sentiment classifier
  • Unity Catalog testing: table ACLs, lineage tracking across the 9-table schema
Sample Databricks prompts
  • Write PySpark to compute average delivery time by seller state
  • Create a Delta table from the orders CSV with schema enforcement
  • Build a window function to rank sellers by monthly revenue
  • Set up an Auto Loader job to ingest new CSVs from ADLS
  • Train a logistic regression model to predict late delivery
No Delta Live Tables streaming demo possible with static CSVs unless you simulate micro-batches or use Databricks Auto Loader with rate limiting.
6
orchestrator
Personal AI Orchestrator
Rich stress test ▼
Why it's the best multi-agent scenario
  • Multiple specialist agents can each own one domain: SQL agent queries orders, NLP agent processes reviews, geo agent maps delivery routes, ML agent scores churn risk
  • Cross-agent data passing: SQL agent extracts seller KPIs → orchestrator routes low performers to an alert agent
  • Human-in-the-loop: orchestrator can pause and prompt "should we flag this seller?" based on review thresholds
  • Tool routing: orchestrator decides whether to call SQL agent or NLP agent based on query intent
Sample orchestration scenario
  • Agent 1 queries top 50 sellers by order volume via SQL
  • Agent 2 computes their average review score from reviews table
  • Agent 3 geo-maps their delivery coverage using geolocation
  • Orchestrator synthesizes: "Seller X has high volume but low reviews in São Paulo — escalate"
Sample orchestrator prompts
  • Find our worst-performing sellers and summarize why customers are unhappy
  • Build a weekly seller health report: volume, revenue, avg review, delivery rate
  • Alert me when any seller's 7-day review average drops below 3.0
Orchestrator complexity may outpace what static data can reveal. You may need to simulate external API calls or tool responses alongside it for a fully rigorous orchestration test.

Advantages & limitations

Advantages

  • Real commercial data — not synthetic
  • Multi-table relational schema (9 tables)
  • Rich join paths via order_id and customer_id
  • Natural language text in review field
  • Time-series built in (2016–2018)
  • Lat/lng geo coordinates included
  • Mixed data types across tables
  • Category translation table (PT → EN)
  • Free — publicly available on Kaggle
  • Joinable with Marketing Funnel dataset

Limitations

  • Static batch — no streaming events
  • Portuguese reviews need translation for some NLP tools
  • No auth / RBAC simulation possible
  • No schema evolution over time
  • 100K rows — won't surface performance bottlenecks
  • No CDC / change data capture support
  • Geolocation has duplicate zip entries (needs dedup)
  • Anonymized data — can't cross-reference externally
Bottom line for Claude Code

This dataset is a self-contained mini data platform. It covers SQL, NLP, geo, ETL, ML, and orchestration — all from one Kaggle download. That makes it an excellent regression harness for all 6 tools in sequence, from easiest (sql-agent) to hardest (orchestrator).

Always provide the schema DDL as context. Load orders first. Join outward. For the orchestrator, consider scripting a lightweight "seller health report" scenario that chains all 6 table domains — it'll exercise every tool capability in a single workflow.
